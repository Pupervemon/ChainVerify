# DeProof 资产通行证 (Asset Passport) 全业务流程指南

## 1. 阶段一：资产数字化（诞生阶段）

这是系统的入口。当一件物理资产（如名表、古董、设备）需要进入数字世界时，流程如下：

1.  **数据采集**: 采集资产的物理指纹（如唯一序列号、哈希）并上传详细图片/文档到 IPFS。
2.  **调用工厂**: 调用 `PassportFactory.createPassport()`。
3.  **原子化执行**:
    *   `AssetPassport` 铸造一个新的 NFT，代表该资产的数字身份。
    *   `ERC6551 Registry` 为该 NFT 部署一个专属的智能合约钱包 (TBA)。
    *   系统将 NFT ID 与 TBA 地址永久绑定。
4.  **结果**: 用户获得了一个 NFT，该 NFT 不仅是所有权凭证，还自带一个可以持有资产、接收资金的钱包账户。

---

## 2. 阶段二：生命周期存证（成长阶段）

资产在使用的过程中会产生各种数据（维修、鉴定、保险、抵押）。这是通过“印章 (Stamp)”体系完成的：

1.  **发起请求**: 某个业务主体（如：认证维修站）尝试为资产添加一条记录。
2.  **调用接口**: 调用 `ChronicleStamp.issueStamp()`。
3.  **权限中转校验**:
    *   `ChronicleStamp` 并不直接判断权限，而是询问 `PassportAuthority`。
    *   `PassportAuthority` 检查该维修站的地址是否在“允许签发维修类印章”的白名单内。
4.  **存证入库**:
    *   校验通过后，`ChronicleStamp` 写入一条不可篡改的 `StampRecord`。
    *   该记录包含了：签发人是谁、发生了什么（元数据 CID）、发生时间。
5.  **记录更新**: 如果是更新旧记录（如：年度检修报告），新印章会标记“替代 (Supersedes)”旧印章 ID，形成清晰的演变链路。

---

## 3. 阶段三：权限治理（管理阶段）

系统管理员（Owner）通过 `PassportAuthority` 来动态调整整个生态的运行规则：

1.  **角色定义**: 管理员定义哪些地址是“工厂（Creators）”、哪些地址是“鉴定师（Issuers）”。
2.  **策略下发**:
    *   **全局模式**: 授权某官方机构可以对所有资产签发所有类型的记录。
    *   **限制模式**: 授权某特定鉴定师只能对“劳力士”类资产签发“鉴定类”印章。
3.  **白名单管理**: 对于极高价值的资产，可以开启 `PassportAllowlistMode`，这意味着只有经过管理员特许的人才能触碰这个资产的存证逻辑。

---

## 4. 阶段四：资产流转（交易阶段）

当所有权发生变更时，系统的优势在于“**数据随所有权打包迁移**”：

1.  **NFT 转让**: 用户 A 将 `AssetPassport` NFT 转让给用户 B（通过 `safeTransferFrom`）。
2.  **档案随迁**: 所有的 `ChronicleStamp` 历史记录自动归属于新持有者查询，不需要进行数据迁移。
3.  **钱包控制权转移**: 由于 TBA (ERC6551) 的控制权是基于“谁持有 NFT”来判定的，用户 B 在获得 NFT 的瞬间，也获得了该资产专属钱包里的所有资金和关联凭证。
4.  **状态变更**: 如果资产发生损坏或被盗，管理员可以将通行证状态设为 `Frozen` 或 `Retired`，强制停止其流转能力。

---

## 5. 业务流程图示

1. **[初始化]** -> Factory 生产 NFT 并生成专用钱包。
2. **[存证]** -> 各类机构申请签发印章，由 Authority 审核身份后由 Stamp 记录。
3. **[查询]** -> 消费者通过 NFT 回溯所有的历史印章，确信资产真伪。
4. **[流转]** -> 所有权转让，历史档案与关联账户自动移交。

---

## 6. 程序员视角的业务精髓

*   **身份即合约**: 每一个资产都是一个活的“智能合约钱包”，而不是数据库里的一行死记录。
*   **插拔式权限**: 业务流程的“逻辑”被锁在 `Authority` 里，而“数据”被存在 `Passport` 和 `Stamp` 里。
*   **全透明溯源**: 任何人（如果被允许）都可以回溯一个资产从生产到报废的每一个环节，且每个环节都有明确的数字签名（Issuer）背书。
