# DeProof 资产通行证 (Asset Passport) 合约部署详解

如果你是第一次接触这个系统的合约部署，或者对 `deploy-passport.ts` 脚本的具体逻辑感到困惑，这份文档将带你从零开始理解整个过程。

---

## 1. 核心概念：为什么要部署四个合约？

我们的资产通行证系统不是一个孤立的合约，而是一个**相互协作的生态系统**。为了实现灵活的权限管理和功能扩展，我们将其拆分为四个部分：

| 合约名称 | 通俗理解 | 主要职责 |
| :--- | :--- | :--- |
| **PassportAuthority** | **权限大脑** | 存储规则：谁有权铸造 NFT？谁有权签发维修记录？ |
| **AssetPassport** | **数字实体** | 本质是一个 NFT (ERC721)，代表资产的所有权和元数据。 |
| **ChronicleStamp** | **溯源档案** | 记录资产的一生：维修、鉴定、转让说明等。 |
| **PassportFactory** | **生产工厂** | 负责“一键创建”：铸造 NFT 的同时，自动为其绑定一个 3D 钱包账户 (TBA)。 |

---

## 2. 部署的“互联 (Wiring)”逻辑

这是最让新人困惑的地方。在区块链上，合约部署后地址是固定的。为了让这四个合约能互相配合，我们需要在部署后执行一套“互联”操作：

1.  **第一步：部署**。脚本会把四个合约代码分别发到链上。此时它们都有了自己的地址，但彼此还不认识。
2.  **第二步：配置引用 (Setting References)**。
    *   告诉 **AssetPassport**：“你的权限校验请找这个 **Authority** 地址”。
    *   告诉 **ChronicleStamp**：“你记录的数据是针对这个 **AssetPassport** 的”。
    *   以此类推，完成全连接。
3.  **第三步：授权**。告诉 **AssetPassport**：“我信任这个 **Factory**，它有权铸造新的通行证”。

**为什么要这么麻烦？** 
因为这样如果你以后想升级“权限规则”，你只需要重新部署一个 `Authority` 并更新一下引用即可，而不需要动到成千上万个已经铸造好的资产 NFT。

## 3. 部署前的配置

脚本会读取 `backend/.env` 里的环境变量。请确保以下项已配置：

```env
# 通行证的名称和符号
PASSPORT_NAME="DeProof Asset Passport"
PASSPORT_SYMBOL="DPAP"

# (核心) 部署完后，谁才是这个系统的最高管理员？
# 如果不填，默认是你用来部署的那个钱包地址
PASSPORT_FINAL_OWNER="0x你的管理员钱包地址"

# (可选) 如果你有现成的 ERC6551 注册表地址
SUBJECT_ACCOUNT_REGISTRY="0x..."
SUBJECT_ACCOUNT_IMPLEMENTATION="0x..."
```

## 4. 实际操作步骤

### 第一步：启动本地测试链
打开一个终端，启动 Hardhat 的本地模拟器：
```bash
cd blockchain
npx hardhat node
```

### 第二步：运行部署脚本
打开另一个终端，执行：
```bash
cd blockchain
# 这一行命令会：编译代码 -> 部署合约 -> 自动完成互联 -> 生成地址清单
npm run deploy:passport:localhost
```

### 第三步：检查产物
部署成功后，你会看到两个重要的变化：
1.  **部署清单**：`blockchain/deployments/passport/localhost.json` 生成了，里面存着所有合约的新地址。
2.  **前端同步**：脚本会自动把这些地址和 ABI (接口定义) 拷贝到 `frontend/src/contracts/passport/`。

## 5. 常见问题 (FAQ)

### Q: 部署脚本提示 `No deployer account found`？
**A**: 检查你的 `.env` 或者 `hardhat.config.ts`。你需要提供一个有测试币的私钥，或者确保本地 `npx hardhat node` 正在运行。

### Q: 部署到公网（如 Sepolia）需要注意什么？
**A**: 
1. 确保你的钱包有 Sepolia ETH。
2. 运行命令改为：`npx hardhat run --network sepolia scripts/deploy-passport.ts`。
3. 部署后，记得在前端检查 `deployment.json` 是否已更新为 sepolia。

### Q: 为什么要“转移所有权 (Transfer Ownership)”？
**A**: 在生产环境下，为了安全，我们通常用一个临时地址部署。部署完并配置好后，我们会把“最高管理权”转交给一个**多签钱包**或者**DAO合约**，防止部署者私自修改规则。

---

## 6. 总结
你只需要运行一次脚本，它就会帮你把：**[代码上链] -> [互相绑定] -> [授权工厂] -> [同步地址到前端]** 这整套流程自动化完成。部署后，你只需要关注 `deployment.json` 里的合约地址即可。
