# DeProof Passport 新人业务权限指南

这份文档只回答新人最容易困惑的三件事：

1. `owner` 到底是谁。
2. 被授权的工厂到底能做什么，不能做什么。
3. `CID` 到底是在存“模板”，还是在存“具体内容”。

## 1. 先说结论

- 当前实现里没有“passport 类型”这个概念。
- 当前链上可配置的“类型”是 `stamp type`，也就是印章类型。
- 创建 passport 不是 “owner 直接铸造”，而是 “有创建资格的人调用工厂，工厂再去铸造”。
- `CID` 大多数时候只是链下内容的引用，不是把模板内容直接写进链上。
- `schemaCID` 更接近“模板/结构定义”，`passportMetadataCID` 和 `assetMetadataCID` 更接近“这本 passport 的具体数据”。

## 2. 先把几种 `owner` 分清

在这套系统里，最容易混淆的是“owner”这个词其实指向不同对象。

| 名称 | 代码里的来源 | 业务含义 |
| --- | --- | --- |
| 合约 owner | `AssetPassport.owner`、`PassportFactory.owner`、`PassportAuthority.owner`、`ChronicleStamp.owner` | 平台级管理员，负责配置系统、授权角色、切换依赖地址 |
| passport 持有人 | `AssetPassport.ownerOf(passportId)` | 某一本 passport NFT 当前属于谁 |
| 工厂 owner | `PassportFactory.owner` | 只表示谁能配置工厂本身，不等于谁能发起创建 passport |
| authority owner | `PassportAuthority.owner` | 权限中心管理员，负责决定谁能创建 passport、谁能签发印章、谁能撤销 |

要特别注意：

- “合约 owner” 和 “passport 持有人” 不是一回事。
- “工厂 owner” 和 “被授权创建 passport 的操作员” 也不是一回事。
- 部署脚本通常会把四个合约的 `owner` 都转给同一个 `finalOwner`，所以在实际部署里它们可能是同一个地址，但概念上仍然是不同职责。

## 3. `owner` 和被授权工厂分别能做什么

这一节只看 `AssetPassport` 主合约，因为“owner vs 工厂”这组问题主要发生在这里。

| 动作 | `AssetPassport` 合约 owner | 已授权工厂地址 | passport 持有人 / ERC721 授权人 | 说明 |
| --- | --- | --- | --- | --- |
| 铸造 passport | 否，除非 owner 地址本身也被加入 factory 白名单 | 是 | 否 | `mintPassport()` 只认工厂地址 |
| 绑定 `subjectAccount` | 是 | 是 | 否 | `setSubjectAccount()` 允许 owner 或工厂调用 |
| 更新 `passportMetadataCID` / `assetMetadataCID` | 是 | 否 | 是 | 持有人、单 token 授权地址、`setApprovalForAll` 操作员都可以 |
| 修改 passport 状态 | 是 | 否 | 否 | 代码里也允许 `authority` 合约地址调用，但当前 `PassportAuthority` 没有转发入口，所以现状基本还是 owner 直接改 |
| 设置 `authority` 地址 | 是 | 否 | 否 | 平台级配置 |
| 设置 `chronicle` 地址 | 是 | 否 | 否 | 平台级配置 |
| 设置 factory 白名单 | 是 | 否 | 否 | 由 owner 决定哪些工厂可信 |
| 转移 passport NFT | 否 | 否 | 是 | 仅持有人或授权人可转，且 passport 必须是 `Active` |

这里最关键的一句是：

> `owner` 在 `AssetPassport` 里主要是“平台配置者”和“兜底管理员”，不是默认的铸造者。

当前实现中，`AssetPassport.mintPassport()` 的权限判断只看 `_factories[operator]`，不看 `owner`。  
也就是说：

- owner 没有工厂授权时，不能直接 mint。
- 工厂有授权时，才能 mint。

## 4. passport 创建其实有两道门

如果只看 `AssetPassport`，你会觉得“只有工厂能 mint”。  
如果只看 `PassportAuthority`，你又会觉得“owner 可以创建 passport”。  
这两句话单看都不完整，真实流程是两道门同时成立。

### 第一道门：业务权限门

外部调用 `PassportFactory.createPassport(initData)` 时，会先走：

`PassportAuthority.canCreatePassport(msg.sender)`

当前规则是：

- `PassportAuthority.owner` 可以创建。
- 被 `setPassportCreator(operator, true)` 授权的地址可以创建。
- 其他地址不能创建。

所以，真正被授权的是“发起创建的人”，不是工厂自己。

### 第二道门：执行权限门

工厂在内部会继续调用：

`AssetPassport.mintPassport(initData.initialHolder, initData)`

但 `AssetPassport` 不信任所有地址，它只信任 owner 事先通过 `setFactory(factory, true)` 放进白名单的工厂合约。

所以，真正被授权的是“执行铸造的工厂合约地址”。

### 合起来看

创建 passport 要同时满足：

1. 调用者被 `PassportAuthority` 认定为可创建者。
2. 被调用的 `PassportFactory` 合约地址被 `AssetPassport` 认定为可信工厂。

可以把它理解成：

- `PassportAuthority` 决定“谁有资格下单”。
- `AssetPassport` 决定“谁有资格动铸造机器”。
- `PassportFactory` 是把这两件事串起来的标准生产线。

## 5. 工厂到底是什么角色

从业务角度看，工厂不是“权限中心”，也不是“最终管理员”，而是一个受控执行器。

它负责的事情只有三步：

1. 接收创建请求。
2. 调用 `AssetPassport` 铸造 passport。
3. 如果配置了 ERC-6551 相关依赖，再创建并绑定 `subjectAccount`。

它不负责的事情包括：

- 决定谁能创建 passport。
- 决定谁能签发印章。
- 决定某本 passport 是否进入 allowlist 模式。
- 决定哪些地址是 stamp type admin。

这些都属于 `PassportAuthority` 或主合约 owner 的职责。

## 6. “owner 可以创建 passport 类型”这个理解为什么不对

当前代码里并没有 `PassportType` 这样的结构，也没有 `createPassportType()` 之类的函数。

`PassportTypes.sol` 这个文件名容易让人误会，但它只是一个 Solidity 类型库，里面放的是：

- `PassportStatus`
- `PassportInitData`
- `PassportRecord`
- `StampTypeConfig`
- `StampRecord`
- `IssuerPolicy`

也就是说，它不是“passport 类型管理器”，只是“这套系统用到的数据结构集合”。

### 当前链上真的可配置的“类型”是什么

当前链上真正可配置的是印章类型，也就是：

- `stampTypeId`
- `code`
- `name`
- `schemaCID`
- `active`
- `singleton`

对应入口是：

`ChronicleStamp.configureStampType(stampTypeId, config)`

谁能配置印章类型：

- `ChronicleStamp.owner`
- 或者被 `PassportAuthority.setStampTypeAdmin(stampTypeId, admin, true)` 授权的类型管理员

所以更准确的说法应该是：

- 不是 “owner 创建 passport 类型”
- 而是 “owner 或被授权的类型管理员配置 stamp type”

## 7. `CID` 到底分别代表什么

这套系统里有好几种 `CID`，它们作用完全不同。

| 字段 | 出现位置 | 业务含义 | 更像模板还是更像实例 |
| --- | --- | --- | --- |
| `passportMetadataCID` | `PassportInitData`、`PassportRecord` | 这本 passport 自己的元数据，比如展示信息、封面、基础说明 | 实例 |
| `assetMetadataCID` | `PassportInitData`、`PassportRecord` | 底层资产的详细资料，比如资产属性、图片、证书详情 | 实例 |
| `schemaCID` | `StampTypeConfig` | 某种印章类型要求遵循的数据结构、字段说明、模板规范 | 模板 / 结构定义 |
| `metadataCID` | `StampRecord` | 某一次具体印章签发的数据内容 | 实例 |
| `policyCID` | `IssuerPolicy` | 链下权限策略说明文档 | 策略说明 |
| `reasonCID` | `revokeStamp()` 参数 | 某次撤销动作对应的原因材料 | 实例 |

### 最容易混淆的一点

创建 passport 时传入的：

- `passportMetadataCID`
- `assetMetadataCID`

它们不是“passport 类型模板”，而是“这本 passport 的具体内容引用”。

真正更接近模板定义的是：

- `schemaCID`

但它属于 `stamp type`，不是 `passport`。

## 8. 一个完整例子

假设你们要给一块手表创建 passport，并允许后续写入维修记录。

### 第一步：平台管理员做系统配置

管理员需要完成这些动作：

1. 给 `AssetPassport` 设置可信工厂：`setFactory(factory, true)`
2. 在 `PassportAuthority` 里授权某个后台账号为创建者：`setPassportCreator(operator, true)`
3. 在 `ChronicleStamp` 里配置一个维修印章类型：
   - `code = "MAINTENANCE"`
   - `name = "维修记录"`
   - `schemaCID = 某份维修记录 JSON Schema 的 CID`
4. 在 `PassportAuthority` 里授权维修服务商可以签发这个印章类型

### 第二步：业务操作员创建 passport

被授权的后台操作员调用：

`PassportFactory.createPassport(initData)`

其中 `initData` 里通常包括：

- `assetFingerprint`: 这块手表的唯一指纹
- `passportMetadataCID`: 这本护照卡片的元数据
- `assetMetadataCID`: 手表资产详情
- `initialHolder`: 初始持有人地址

### 第三步：工厂完成标准化执行

工厂会自动做：

1. 调 `AssetPassport.mintPassport(...)`
2. 如已配置 ERC-6551，创建 `subjectAccount`
3. 调 `AssetPassport.setSubjectAccount(...)` 绑定主体账户

### 第四步：后续写入维修记录

维修服务商不能去改 passport 类型，它做的是：

1. 按照 `schemaCID` 对应的模板整理维修数据
2. 把这次维修记录上传到 IPFS / Arweave，得到 `metadataCID`
3. 调用 `ChronicleStamp.issueStamp(...)`

也就是说：

- 模板属于 `stamp type`
- 实际数据属于某条具体 `stamp`

## 9. 新人最容易踩的坑

### 误区 1：owner 一定可以直接 mint

不对。  
当前实现中，`AssetPassport.owner` 不自动拥有 mint 权限。

### 误区 2：工厂 owner 就等于可以创建 passport

不对。  
`createPassport()` 不看 `PassportFactory.owner`，而是看 `PassportAuthority.canCreatePassport(msg.sender)`。

### 误区 3：`PassportTypes.sol` 就是在管理 passport 类型

不对。  
它只是类型定义库，不是“passport 分类系统”。

### 误区 4：创建 passport 时传的 CID 是类型模板

不对。  
创建时传的两个 CID 都是“实例内容”。  
真正像模板的是 `stamp type` 里的 `schemaCID`。

### 误区 5：`authority` owner 一定能直接冻结 passport

当前实现里也不完全对。  
`AssetPassport.setPassportStatus()` 允许 `owner` 或 `authority` 合约地址调用，但现在 `PassportAuthority` 没有提供转发这个动作的外部函数，所以现状通常仍由 `AssetPassport.owner` 直接改状态。

## 10. 最后用一句话记住这套权限模型

如果只记一句话，可以记这个：

> `owner` 负责配置和授权，工厂负责标准化执行，passport 持有人负责持有和部分更新，`stamp type` 才是模板层，passport 创建时的 CID 只是具体内容引用。
